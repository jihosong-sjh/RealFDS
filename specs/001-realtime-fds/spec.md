# Feature Specification: 실시간 금융 거래 탐지 시스템 (Real-time Financial Detection System)

**Feature Branch**: `001-realtime-fds`
**Created**: 2025-11-05
**Status**: Draft
**Input**: 실시간으로 금융 거래를 모니터링하여 의심스러운 패턴을 즉시 탐지하고 보안 담당자에게 알림을 제공하는 시스템

## 비전 및 목적

### 무엇을 만드는가
실시간으로 금융 거래를 모니터링하여 의심스러운 패턴을 즉시 탐지하고 보안 담당자에게 알림을 제공하는 시스템

### 왜 만드는가
- **학습 목표**: 대규모 실시간 데이터 스트리밍 처리 기술 습득
- **포트폴리오 목표**: 이벤트 기반 마이크로서비스 아키텍처 구현 능력 증명
- **비즈니스 맥락**: 금융 사기는 즉각적인 대응이 필요하며, 사후 탐지는 이미 늦음

### 성공 지표
- 거래 발생부터 알림 표시까지 **5초 이내**
- 시스템 가용성 **99%** 이상
- 단일 명령어로 전체 시스템 실행 가능

## 사용자 및 페르소나

### 주요 사용자: 금융 보안 담당자 (Security Analyst)
- **이름**: 김보안 (30세, 3년 경력)
- **목표**:
  - 의심 거래를 실시간으로 모니터링
  - 패턴 기반 자동 탐지로 업무 부담 감소
  - 빠른 대응으로 피해 최소화
- **문제점**:
  - 수동 모니터링으로 놓치는 거래 많음
  - 사후 발견 시 이미 피해 발생
  - 복잡한 시스템은 사용하기 어려움
- **기술 수준**: 웹 브라우저 사용 가능, 기술적 배경 없음

### 부차적 사용자: 시스템 개발자
- **목표**:
  - 새로운 탐지 규칙을 쉽게 추가
  - 시스템을 로컬에서 빠르게 테스트
- **문제점**:
  - 복잡한 설정으로 시작이 어려움
  - 디버깅이 어려운 분산 시스템

## User Scenarios & Testing

### User Story 1 - 실시간 알림 모니터링 (Priority: P1)

금융 보안 담당자가 웹 브라우저를 통해 의심 거래 알림을 실시간으로 모니터링하고 즉시 대응할 수 있는 기능

**Why this priority**: 시스템의 핵심 가치 제안이며, 사용자가 가장 먼저 경험하게 되는 주요 기능. 이 기능 없이는 시스템의 존재 이유가 없음.

**Independent Test**: 웹 브라우저를 열고 대시보드에 접속하여 실시간으로 알림이 표시되는지 확인. 알림 생성 시 1초 이내에 화면에 나타나는지 측정하여 독립적으로 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 보안 담당자가 웹 브라우저를 실행하고, **When** 알림 대시보드 URL에 접속하면, **Then** 자동으로 실시간 연결이 수립되고 화면에 "실시간 연결됨" 상태가 표시됨

2. **Given** 대시보드가 연결된 상태에서, **When** 새로운 의심 거래가 탐지되면, **Then** 1초 이내에 화면에 알림이 표시되며, 발생 시각, 거래 ID, 사용자 ID, 거래 금액, 탐지 규칙, 상세 사유가 포함됨

3. **Given** 알림 목록에 100개 이상의 알림이 있을 때, **When** 새로운 알림이 추가되면, **Then** 가장 오래된 알림이 제거되고 최근 100개만 유지됨

4. **Given** 대시보드가 연결되어 있다가, **When** 네트워크 문제로 연결이 끊어지면, **Then** 화면에 "연결 끊김" 상태가 표시되고 자동으로 재연결을 시도함

5. **Given** 연결이 끊긴 상태에서, **When** 네트워크가 복구되면, **Then** 자동으로 재연결되고 실시간 스트리밍이 재개됨

---

### User Story 2 - 고액 거래 자동 탐지 (Priority: P2)

시스템이 100만원을 초과하는 거래를 자동으로 탐지하여 보안 담당자에게 알림을 제공하는 기능

**Why this priority**: 첫 번째 구체적인 탐지 규칙으로, 가장 단순하고 명확한 규칙. 이 기능만으로도 고액 거래에 대한 모니터링 가치를 제공할 수 있음.

**Independent Test**: 테스트 거래 데이터를 생성하여 100만원 초과 거래를 발생시키고, 5초 이내에 알림이 표시되는지 확인. 알림에 정확한 금액과 "고액 거래" 사유가 표시되는지 검증.

**Acceptance Scenarios**:

1. **Given** 시스템이 실행 중일 때, **When** 거래 금액이 1,000,000원을 초과하는 거래가 발생하면, **Then** 알림이 생성되고 "고액 거래 (100만원 초과)"라는 사유와 함께 정확한 금액이 표시됨

2. **Given** 고액 거래가 발생했을 때, **When** 알림이 생성되면, **Then** 거래 발생 후 5초 이내에 대시보드 화면에 표시됨

3. **Given** 거래 금액이 정확히 1,000,000원일 때, **When** 거래가 발생하면, **Then** 알림이 생성되지 않음 (초과가 아니므로)

4. **Given** 여러 개의 고액 거래가 연속으로 발생할 때, **When** 각 거래가 처리되면, **Then** 각 거래에 대해 독립적인 알림이 생성됨

---

### User Story 3 - 해외 거래 자동 탐지 (Priority: P2)

시스템이 한국 외 지역에서 발생한 거래를 탐지하여 비정상적인 해외 거래 패턴을 빠르게 인지하는 기능

**Why this priority**: 고액 거래 탐지와 동일한 우선순위의 단순 규칙. 지리적 위치 기반 탐지는 사기 탐지의 기본 패턴 중 하나.

**Independent Test**: 다양한 국가 코드(US, JP, CN)를 가진 거래를 생성하고, KR이 아닌 모든 거래에 대해 알림이 발생하는지 확인. 알림에 국가 코드가 명시되는지 검증.

**Acceptance Scenarios**:

1. **Given** 시스템이 실행 중일 때, **When** 국가 코드가 "KR"이 아닌 거래가 발생하면, **Then** "해외 거래 탐지 (국가: {국가코드})" 사유와 함께 알림이 생성됨

2. **Given** 국가 코드가 "KR"인 거래가 발생할 때, **When** 거래가 처리되면, **Then** 해외 거래 알림이 생성되지 않음

3. **Given** 해외 거래가 발생했을 때, **When** 알림이 생성되면, **Then** 알림에 거래가 발생한 정확한 국가 코드가 표시됨

4. **Given** 동일한 사용자가 여러 국가에서 거래를 발생시킬 때, **When** 각 해외 거래가 발생하면, **Then** 각 국가별로 독립적인 알림이 생성됨

---

### User Story 4 - 빈번한 거래 패턴 탐지 (Priority: P3)

시스템이 동일 사용자가 짧은 시간(1분)에 여러 번(5회 초과) 거래하는 패턴을 탐지하여 자동화된 사기 시도를 차단하는 기능

**Why this priority**: 가장 복잡한 탐지 규칙으로, 시간 윈도우와 상태 추적이 필요. 앞선 두 규칙이 작동한 후에 구현해도 충분한 가치 제공 가능.

**Independent Test**: 특정 사용자의 거래를 1분 내에 6회 생성하고, 6번째 거래부터 알림이 발생하는지 확인. 1분 경과 후 카운터가 리셋되는지 시간 기반 테스트 수행.

**Acceptance Scenarios**:

1. **Given** 특정 사용자가 거래를 시작할 때, **When** 최근 1분 이내에 5회를 초과하는 거래가 발생하면, **Then** 6번째 거래부터 "빈번한 거래 (1분 내 5회 초과)" 사유와 함께 알림이 생성됨

2. **Given** 특정 사용자가 1분 내에 5회 이하의 거래를 했을 때, **When** 거래가 처리되면, **Then** 빈번한 거래 알림이 생성되지 않음

3. **Given** 사용자가 1분 내에 5회 거래를 하고 1분이 경과한 후, **When** 새로운 거래가 발생하면, **Then** 카운터가 초기화되어 알림이 생성되지 않음

4. **Given** 여러 사용자가 동시에 거래할 때, **When** 각 사용자의 거래 패턴이 평가되면, **Then** 각 사용자별로 독립적으로 카운터가 추적됨

5. **Given** 사용자가 1분 내에 6회 이상 거래를 계속할 때, **When** 6번째 이후의 거래가 발생하면, **Then** 각 거래마다 알림이 계속 생성됨

---

### User Story 5 - 간편한 시스템 실행 (Priority: P1)

개발자 또는 데모 진행자가 단일 명령어로 전체 시스템을 실행하고 종료할 수 있는 기능

**Why this priority**: 시스템 데모와 개발 효율성의 핵심. 복잡한 설정 없이 빠르게 시스템을 시작할 수 있어야 학습 및 포트폴리오 목적을 달성 가능.

**Independent Test**: 프로젝트 디렉토리에서 단일 명령어를 실행하고, 5분 이내에 모든 구성 요소가 정상 작동하는지 확인. 시스템 종료도 단일 명령어로 가능한지 테스트.

**Acceptance Scenarios**:

1. **Given** Docker와 Docker Compose가 설치된 환경에서, **When** `docker-compose up` 명령을 실행하면, **Then** 5분 이내에 모든 구성 요소(데이터 생성기, 탐지 엔진, 알림 서비스, 웹 대시보드)가 실행됨

2. **Given** 시스템이 실행 중일 때, **When** 각 구성 요소의 로그를 확인하면, **Then** 명확한 상태 메시지(예: "서비스 시작됨", "연결 성공")가 표시됨

3. **Given** 시스템이 실행 중일 때, **When** `docker-compose down` 명령을 실행하면, **Then** 모든 구성 요소가 정상적으로 종료됨

4. **Given** 시스템 시작 후, **When** 웹 브라우저에서 대시보드에 접속하면, **Then** 추가 설정 없이 즉시 실시간 알림 스트리밍이 작동함

5. **Given** 시스템이 30분간 실행 중일 때, **When** 모니터링하면, **Then** 크래시나 중단 없이 안정적으로 작동함

### Edge Cases

- **네트워크 지연**: 알림 전송 중 네트워크 지연이 발생하면 어떻게 되는가? 알림은 버퍼링되고 연결 복구 시 전송되어야 함.
- **동시 다중 알림**: 짧은 시간에 수십 개의 알림이 동시에 발생하면 화면이 어떻게 처리하는가? 최근 100개만 유지하고 나머지는 제거.
- **잘못된 거래 데이터**: 금액이 음수이거나 국가 코드가 유효하지 않은 경우 시스템은 어떻게 처리하는가? 로그에 경고를 기록하고 해당 거래는 무시.
- **시스템 시작 시 초기 상태**: 시스템 시작 직후 첫 알림이 나타나기까지 시간이 얼마나 걸리는가? 데이터 생성 주기에 따라 다르지만 최대 5분 이내.
- **브라우저 탭 전환**: 사용자가 대시보드 탭에서 다른 탭으로 전환했다가 돌아오면 실시간 연결은 유지되는가? WebSocket 연결은 유지되며, 탭 전환 중에도 알림은 계속 수신됨.
- **시스템 재시작**: 시스템 재시작 시 이전 알림 이력은 어떻게 되는가? 이 버전에서는 알림 이력을 저장하지 않으므로 재시작 시 모든 알림이 초기화됨 (Out of Scope에 명시됨).
- **메모리 부족**: 로컬 환경에서 메모리가 부족하면 어떻게 되는가? README에 최소 사양(8GB RAM)을 명시하고, 메모리 부족 시 컨테이너가 재시작될 수 있음을 경고.

## Requirements

### Functional Requirements

#### 데이터 생성
- **FR-001**: 시스템은 테스트 및 데모를 위해 가상의 거래 데이터를 주기적으로 자동 생성해야 함
- **FR-002**: 각 거래는 고유한 거래 ID, 사용자 ID, 거래 금액, 통화, 국가 코드, 거래 발생 시각을 포함해야 함
- **FR-003**: 사용자 ID는 10명의 가상 사용자 풀에서 무작위로 선택되어야 함
- **FR-004**: 거래 금액은 1,000원에서 1,500,000원 사이의 무작위 값이어야 함
- **FR-005**: 통화는 항상 "KRW"이어야 함
- **FR-006**: 국가 코드는 대부분 "KR"이어야 하며, 가끔 "US", "JP", "CN"이 포함되어야 함 (데모 효과를 위해)

#### 실시간 탐지
- **FR-007**: 시스템은 새로운 거래가 발생하면 즉시 3가지 탐지 규칙(고액 거래, 해외 거래, 빈번한 거래)을 적용해야 함
- **FR-008**: 거래 금액이 1,000,000원을 초과하면 "고액 거래 (100만원 초과)" 사유와 함께 알림을 생성해야 함
- **FR-009**: 거래 국가 코드가 "KR"이 아니면 "해외 거래 탐지 (국가: {국가코드})" 사유와 함께 알림을 생성해야 함
- **FR-010**: 동일 사용자가 최근 1분 이내에 5회를 초과하는 거래를 발생시키면 6번째 거래부터 "빈번한 거래 (1분 내 5회 초과)" 사유와 함께 알림을 생성해야 함
- **FR-011**: 시스템은 각 사용자별로 독립적으로 거래 빈도를 추적해야 함
- **FR-012**: 1분이 지나면 해당 사용자의 거래 빈도 카운터는 자동으로 초기화되어야 함
- **FR-013**: 하나의 거래가 여러 탐지 규칙에 해당하는 경우, 각 규칙별로 독립적인 알림을 생성해야 함

#### 실시간 알림 표시
- **FR-014**: 웹 대시보드는 사용자가 접속하면 자동으로 실시간 연결을 수립해야 함
- **FR-015**: 화면에는 현재 연결 상태("실시간 연결됨" 또는 "연결 끊김")가 명확히 표시되어야 함
- **FR-016**: 새로운 알림이 탐지되면 1초 이내에 화면에 표시되어야 함
- **FR-017**: 각 알림에는 발생 시각, 거래 ID, 사용자 ID, 거래 금액, 탐지 규칙, 상세 사유가 포함되어야 함
- **FR-018**: 새로운 알림은 목록의 맨 위에 추가되어야 함 (최신 알림이 가장 위)
- **FR-019**: 화면에는 최근 100개의 알림만 유지되어야 하며, 101번째 알림이 추가되면 가장 오래된 알림이 제거되어야 함
- **FR-020**: 네트워크 문제로 연결이 끊어지면 자동으로 재연결을 시도해야 함
- **FR-021**: 재연결 성공 시 실시간 스트리밍이 자동으로 재개되어야 함
- **FR-022**: 대시보드는 별도의 로그인이나 설정 없이 즉시 사용 가능해야 함

#### 시스템 실행 및 운영
- **FR-023**: 단일 명령어(`docker-compose up`)로 모든 구성 요소가 실행되어야 함
- **FR-024**: 시스템은 5분 이내에 완전히 준비 상태가 되어야 함
- **FR-025**: 각 구성 요소는 시작 시 명확한 상태 메시지를 로그에 출력해야 함
- **FR-026**: 단일 명령어(`docker-compose down`)로 모든 구성 요소가 정상 종료되어야 함
- **FR-027**: 시스템은 30분간 중단 없이 안정적으로 실행되어야 함

### Key Entities

- **Transaction (거래)**: 금융 거래를 나타내는 데이터 엔터티
  - 속성: 거래 ID (고유 식별자), 사용자 ID, 거래 금액, 통화, 국가 코드, 거래 발생 시각
  - 역할: 시스템에서 탐지 규칙이 적용되는 기본 단위

- **Alert (알림)**: 탐지된 의심스러운 패턴을 나타내는 엔터티
  - 속성: 알림 ID, 발생 시각, 거래 ID (참조), 사용자 ID, 거래 금액, 탐지 규칙명, 상세 사유
  - 역할: 보안 담당자에게 전달되는 실시간 경고 정보

- **User (사용자)**: 거래를 발생시키는 가상 사용자
  - 속성: 사용자 ID (고유 식별자), 최근 거래 시각 목록 (빈번한 거래 탐지를 위한 임시 저장)
  - 역할: 거래 발생 주체이며, 빈번한 거래 패턴 탐지 시 추적 대상

- **Detection Rule (탐지 규칙)**: 의심 패턴을 정의하는 비즈니스 로직
  - 속성: 규칙명, 규칙 설명, 탐지 조건
  - 역할: 거래 데이터를 평가하여 알림 생성 여부를 결정
  - 예시: "고액 거래", "해외 거래", "빈번한 거래"

## Success Criteria

### Measurable Outcomes

- **SC-001**: 거래가 발생한 시점부터 웹 대시보드에 알림이 표시되기까지 평균 3초, 최대 5초 이내
- **SC-002**: 시스템은 초당 최소 10개의 거래를 처리할 수 있어야 함
- **SC-003**: 최소 10명의 사용자가 동시에 대시보드에 접속하여 실시간 알림을 볼 수 있어야 함
- **SC-004**: 시스템은 30분간 크래시나 중단 없이 안정적으로 실행되어야 함 (가용성 목표)
- **SC-005**: 단일 명령어 실행 후 5분 이내에 시스템이 완전히 준비되어 웹 대시보드에 접속 가능해야 함
- **SC-006**: 네트워크 일시 장애 후 복구 시, 시스템은 수동 개입 없이 자동으로 정상 작동해야 함
- **SC-007**: 사전 기술 지식이 없는 사용자가 README를 읽고 10분 내에 시스템을 실행할 수 있어야 함
- **SC-008**: 보안 담당자가 별도의 사용 설명서 없이 대시보드의 기능을 직관적으로 이해하고 사용할 수 있어야 함
- **SC-009**: 고액 거래 탐지 규칙은 100% 정확도로 작동해야 함 (100만원 초과 거래 모두 탐지)
- **SC-010**: 해외 거래 탐지 규칙은 100% 정확도로 작동해야 함 (KR 외 국가 거래 모두 탐지)
- **SC-011**: 빈번한 거래 탐지 규칙은 1분 내 5회 초과 거래에 대해 100% 정확도로 작동해야 함
- **SC-012**: 새로운 탐지 규칙을 추가하는 데 개발자가 30분 이내에 완료할 수 있어야 함 (유지보수성)

## 제약사항 및 가정

### Technical Constraints
**필수 요구사항**:
- 시스템은 **컨테이너 기반**으로 구성되어야 함 (환경 일관성 보장)
- 서비스 간 통신은 **비동기 메시징**을 사용해야 함 (확장성 및 결합도 감소)
- **시간 기반 윈도우 처리**가 가능해야 함 (빈번한 거래 탐지를 위해)
- 실시간 양방향 통신 지원 (웹소켓 등)

**선호 사항** (학습 목적):
- 대규모 스트리밍 처리에 적합한 메시지 브로커 활용
- 상태 저장(stateful) 스트림 처리 가능한 프레임워크 사용
- 이벤트 기반 마이크로서비스 아키텍처 패턴 적용

### Environmental Constraints
- 로컬 개발 환경에서 실행 (최소 8GB RAM, 4 core CPU)
- Docker와 Docker Compose 설치 필수
- 인터넷 연결 필요 (컨테이너 이미지 다운로드)
- Windows, macOS, Linux에서 동일하게 작동해야 함

### Assumptions
- 사용자는 웹 브라우저(Chrome, Firefox, Edge 등 최신 버전)를 사용할 수 있음
- 사용자는 터미널에서 기본 명령어(`cd`, `docker-compose up`)를 실행할 수 있음
- 이 시스템은 학습 및 데모 목적이며, 실제 운영 환경에서는 추가 보안, 확장성, 데이터 영속성 고려 필요
- 알림 데이터는 메모리에만 저장되며, 시스템 재시작 시 초기화됨
- 거래 데이터는 실제 금융 데이터가 아닌 무작위 생성된 가상 데이터임
- 네트워크 지연은 일반적인 로컬 환경 수준(수 밀리초 이내)이라고 가정

## 향후 개선 사항 (현재 범위 외)

현재 버전에서는 구현하지 않지만, 향후 추가를 고려할 수 있는 기능:

1. **과거 알림 조회**: 지난 알림을 검색하고 필터링하는 기능
2. **알림 통계**: 시간대별, 규칙별 알림 발생 추이를 차트로 시각화
3. **동적 규칙 관리**: 시스템 재시작 없이 웹 UI에서 새로운 탐지 규칙 추가/수정/삭제
4. **사용자 인증**: 여러 보안 담당자가 개별 계정으로 접속하고 역할 기반 접근 제어
5. **모바일 지원**: 스마트폰에서도 알림을 확인할 수 있는 반응형 UI 또는 모바일 앱
6. **머신러닝 탐지**: 과거 데이터를 학습하여 새로운 사기 패턴을 자동으로 탐지
7. **알림 확인/처리**: 담당자가 알림을 확인했음을 표시하고, 조치 결과를 기록
8. **실시간 대시보드**: 거래량, 알림 발생률, 시스템 상태를 차트와 그래프로 시각화
9. **알림 우선순위**: 탐지 규칙별로 알림 심각도를 설정하고, 중요도에 따라 색상이나 아이콘으로 구분
10. **다국어 지원**: 영어, 한국어 등 여러 언어로 UI 제공

## 데모 시나리오

시스템 완성 후 다음 시나리오로 데모 진행:

### 데모 준비
1. 발표자의 노트북에 Docker와 Docker Compose가 설치되어 있음
2. 프로젝트 폴더로 이동
3. README를 참고하여 사전 준비사항 확인 (네트워크 연결, 포트 가용성 등)

### 데모 실행
1. **시스템 시작**: `docker-compose up` 명령 실행
   - 5분 이내에 모든 구성 요소(메시지 브로커, 데이터 생성기, 탐지 엔진, 알림 서비스, 웹 대시보드) 실행 확인
   - 각 서비스의 시작 로그가 화면에 출력됨

2. **대시보드 접속**: 웹 브라우저에서 `http://localhost:8083` 접속
   - "실시간 FDS 알림" 제목의 페이지 표시
   - "실시간 연결됨" 상태 표시 확인

3. **실시간 알림 관찰**: 화면을 관찰하며 다음 알림들이 실시간으로 나타나는지 확인
   - 고액 거래 알림: 거래 금액이 150만원인 거래 발생 시
   - 해외 거래 알림: 미국(US) 또는 일본(JP)에서 발생한 거래
   - 빈번한 거래 알림: 특정 사용자가 1분 내에 6번째 거래를 발생시킬 때

4. **성능 확인**: 각 알림이 거래 발생 후 1-3초 이내에 화면에 표시되는지 확인

5. **안정성 확인**: 시스템을 5분간 실행하며 크래시나 오류 없이 계속 작동하는지 확인

6. **시스템 종료**: `docker-compose down` 명령으로 모든 서비스 정상 종료

- **단순성**: 단일 명령어로 전체 시스템 실행
- **실시간성**: 거래 발생 즉시 알림 표시
- **다양성**: 3가지 탐지 규칙이 모두 작동하는 모습 시연
- **안정성**: 지속적으로 안정적으로 작동하는 시스템
